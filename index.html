The Web Accessibility Directive

European-union

Está em vigor desde 22 de dezembro de 2016 e oferece às pessoas com deficiência um melhor acesso a sites e aplicativos de serviços públicos.

Com isso, a União Européia está exigindo oficialmente que seus estados membros cumpram os padrões WCAG 2.1 Nível AA.

Tipos de deficiência

Seguir estas diretrizes irá tornar o conteúdo acessível a um maior número de pessoas com deficiência, incluindo:

  • Cegueira e baixa visão
  • Limitações cognitivas
  • Limitações de movimentos
  • Fotossensibilidade e combinações destas características
  • Dificuldades de aprendizagem
  • Surdez e baixa audição
  • Incapacidade de fala

ARIA

Accessible Rich Internet Applications

Sem ARIA, certas funcionalidades ficam indisponíveis para alguns usuários com deficiências, especialmente pessoas que dependem de leitores de tela ou que não podem usar um mouse.

ARIA não deve ser usado quando o HTML já fornece semântica suficiente para acessibilidade

ARIA Roles

  • Definem o que um elemento é ou faz
  • Definem funcionalidades que não estão disponíveis no HTML
  • Substituem determinadas funções-padrão dos elementos HTML
                                
    <div role="button" id="saveChanges" tabindex="0" aria-pressed="false">Save</div>
    
    <!-- Equivalente a -->
    
    <button type="button" id="saveChanges">Save</button>

WAI-ARIA Roles

ARIA States

São atributos que definem a condição atual de um elemento.

Geralmente mudam com base em alguma interação feita.

Um exemplo é:

                                
    <!-- Menu -->
    <button aria-expanded="false">Menu</button>

ARIA states and properties

ARIA Properties

Definem semânticas adicionais não suportadas no HTML padrão.
Um exemplo é:

                                
    <!-- aria-haspopup -->
    <button aria-haspopup="true">

Essa propriedade faz com que um leitor de tela anuncie que o botão, se ativado, acionará um pop-up.



Outro exemplo:

                                
    <!-- aria-controls -->
    <button role="tab" aria-controls="panel-1" aria-selected="true">First Tab</button>

aria-controls identifica o elemento (ou elementos) cujo conteúdo ou presença são controlados pelo elemento no qual este atributo é definido

Web Content Accessibility Guidelines (WCAG) 2.1

Conteúdo Não Textual

Fornecer alternativas textuais para qualquer conteúdo não textual

  • Controles / Inputs: Se o conteúdo não textual for um controle ou input, ele deve ter um nome que descreve sua finalidade 
                                        
<!-- Button -->
<button aria-label="Close">x</button>

<!-- Input -->
<label for="input-1">Input description</label>
<input type="text" id="input-1">
  • Mídias: Devem conter uma identificação descritiva do conteúdo não textual 
                                            
<img src="girl.jpg" alt="Girl in a jacket">
  • Decoração, Formatação, Invisível: Se o conteúdo for decorativo, ou se não for exibido aos usuários, deve ser ignorado pelas tecnologias assistivas 
                                            
<a href="#">
    <img src="icon.svg" aria-hidden="true">
    <span>Link text</span>
</a>

Success Criterion 1.1.1 Non-text Content

Reflow

O conteúdo pode ser apresentado sem perda de informação ou funcionalidade e sem exigir scroll horizontal com zoom de até 400%

Success Criterion 1.4.10 Reflow

Espaçamento de Texto

Não pode haver nenhuma perda de conteúdo ou funcionalidade pela configuração manual de todos os itens a seguir:

  • Altura da linha (espaçamento entre linhas) de pelo menos 1,5 vezes o tamanho da fonte
  • Espaçamento entre os parágrafos subsequentes de pelo menos 2 vezes o tamanho da fonte
  • Espaçamento entre letras de pelo menos 0,12 vezes o tamanho da fonte
  • Espaçamento entre palavras de pelo menos 0,16 vezes o tamanho da fonte

O objetivo desta técnica é permitir que um usuário altere o espaçamento do texto por meio do CSS, bookmarklets, extensões ou aplicativos.

Nunca devemos especificar a propriedade "height" em contêineres de texto.
Isso evita erros caso o texto se expanda verticalmente.

Success Criterion 1.4.12 Text Spacing

Allowing for text spacing override

CSS example
                    
/* rules for bookmarklet test */

* {
    line-height: 1.5 !important;
    letter-spacing: 0.12em !important;
    word-spacing: 0.16em !important;
}

p{
    margin-bottom: 2em !important;
}

Conteúdo em foco por mouse ou teclado

Quando o hover do mouse, ou o focus do teclado exibem algum conteúdo, os itens a seguir devem ser respeitados:

  • Descartável: É possível ocultar o conteúdo adicional sem mover o cursor ou o foco do teclado, a menos que o conteúdo adicional comunique um erro de input ou não oculte ou substitua outro conteúdo (ESC, por exemplo);
  • "Hoverable" Se passar o cursor sobre o conteúdo adicional o ativa, então o cursor poderá ser movido sobre o conteúdo adicional sem que o conteúdo desapareça (tooltips, por exemplo);
  • Persistente: O conteúdo adicional permanece visível até que o over ou focus do teclado seja removido, o usuário o descarte ou se a informação não seja mais válida.

Tooltips personalizados, submenus e outros popups não modais exibidos ao passar o cursor sobre e com o foco são exemplos de conteúdos adicionais cobertos por esse critério.

Success Criterion 1.4.13 Content on Hover or Focus

Acessível por Teclado

Fazer com que toda funcionalidade fique disponível a partir de um teclado

Critério de Sucesso 2.1.2 - No Keyboard Trap (A)

Se o foco puder ser movido para um componente utilizando um teclado, então o foco pode ser retirado desse componente utilizando apenas o teclado e, se for necessário mais do que as setas, tab ou outros métodos de saída normalmente utilizados, o usuário deve ser informado sobre o método para retirar o foco.



Critério de Sucesso 2.1.3 - Teclado Sem Exceção (AAA)

Toda a funcionalidade é acessível através de um teclado.

Guideline 2.1 Keyboard Accessible

Tempo Suficiente

Fornecer aos usuários tempo suficiente para ler e utilizar o conteúdo.

Critério de Sucesso 2.2.2 Colocar em Pausa, Parar, Ocultar (A)


  • Moving, blinking, scrolling: Para quaisquer informações em movimento, em modo intermitente ou em deslocamento, que (1) sejam iniciadas automaticamente, (2) durem mais de cinco segundos, e (3) sejam apresentadas em paralelo com outro conteúdo, existe um mecanismo para o usuário colocar em pausa, parar, ou ocultar as mesmas, a menos que este movimento faça parte de uma atividade na qual seja essencial;
  • Auto-updating: O mesmo vale pare seções onde o conteúdo é atualizado automaticamente
Exemplos: Vídeos, sliders, animações, jogos, áudio, publicidade, informações meteorológicas atualizadas automaticamente, notícias, atualizações de preços de ações, etc.

Guideline 2.2 Enough Time

Pause, Stop, Hide (Level A)

Convulsões e Reações Físicas

Não criar conteúdo de uma forma conhecida por causar convulsões e reações físicas.

Critério de Sucesso 2.3.2 Três Flashes (AAA)

As páginas web não incluem qualquer conteúdo que pisca mais de três vezes no período de um segundo.



Critério de Sucesso 2.3.3 Animação de Interações (AAA)

A animação de movimento acionada por interação pode ser desativada, a menos que a animação seja essencial para a funcionalidade ou para as informações que estão sendo transmitidas.

Utilizar a Media Query prefers-reduced-motion
Usuários com distúrbios de movimento vestibular (tontura provocada pelo movimento, labirintite, por exemplo) podem desejar ativar o recurso de redução de movimento diretamente em seu dispositivo

Guideline 2.3 Seizures and Physical Reactions

Navegável

Fornecer maneiras de ajudar os usuários a navegar, localizar conteúdos e determinar onde se encontram.

Critério de Sucesso 2.4.1 - Ignorar Blocos (A)

Um mecanismo está disponível para ignorar blocos de conteúdo que são repetidos em várias páginas web.

Criar um link Skip to main no topo da página, que vai direto para a seção main
Exemplo: https://pt.wikipedia.org. Aperte "Tab" assim que a página for carregada.


Critério de Sucesso 2.4.3 - Ordem de Foco

Se a sequência de navegação afetar o significado ou a operação, os componentes que podem ser focados recebem o foco em uma ordem que preserva o significado e a operabilidade.

Como resolver: Utilizar atributo tabindex quando necessário.


Critério de Sucesso 2.4.4 - Finalidade do Link Em contexto (A)

A finalidade de cada link pode ser determinada a partir do link sozinho ou a partir do texto do link em conjunto com seu respectivo contexto do link determinado por meio de código de programação,exceto quando a finalidade do link for ambígua para os usuários em geral.

Como resolver: Se houver um título que descreva a funcionalidade, usar atributo aria-labelledby;
Se não, adicionar a descrição com o atributo aria-label.


Critério de Sucesso 2.4.7 - Foco Visível (AA)

Todos os elementos que podem ser alcançados pelo teclado, devem ter um indicador de foco visível.

Manter o indicador padrão de :focus / :focus-visible do browser, ou adicionar manualmente em elementos customizados.


Critério de Sucesso 2.4.10 - Section Headings (AAA)

Utilizados para organizar o conteúdo na página.

Organizar o conteúdo utilizando as tags corretas, respeitando a hierarquia <h1> , <h2> , <h3> ...

Guideline 2.4 Navigable

Tamanho do Alvo (Área Clicável) (AAA)

O tamanho de alvos clicáveis deve ser de pel menos 44px por 44px, exceto quando:


  • Equivalente: O alvo está disponível por meio de um outro link ou botão equivalente na mesma página, com pelo menos 44 por 44 pixels;
  • Inline: O alvo está em uma sentença ou bloco de texto;
  • User Agent Control: O tamanho do alvo é determinado pelo "User Agent" e não é modificado pelo autor;
  • Essencial: Uma apresentação específica do alvo é essencial para as informações que estão sendo transmitidas.
